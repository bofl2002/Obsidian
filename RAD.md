Rapid Application Development

[[Typer af Systemudv. metoder]]


**RAD (Rapid Application Development)** er en systemudviklingsmetode, der fokuserer på **hurtig udvikling** gennem iterative prototyper og tæt brugerinvolvering. Metoden blev udviklet i 1980'erne som et alternativ til den langsomme traditionelle vandfaldsmodel.

Hovedprincipper

- **Hurtig udvikling** - Kort udviklingstid (typisk 60-90 dage)
- **Prototyping** - Udvikling gennem iterative prototyper
- **Brugerinvolvering** - Brugere deltager aktivt gennem hele processen
- **Fleksibilitet** - Kan tilpasse sig ændrede krav
- **Mindre dokumentation** - Fokus på funktionelt system frem for omfattende dokumentation
- **Genbrugelige komponenter** - Brug af eksisterende kode og værktøjer

---

De 4 faser i RAD

1. Requirements Planning (Kravplanlægning)

Identificer forretningsmål og krav på højt niveau
Definér projektets omfang
Involvér nøglebrugere og stakeholders
Kort fase - fokus på overordnede behov

2. User Design (Brugerdesign)

Tæt samarbejde mellem brugere og udviklere
Udvikling af prototyper
Brugere giver feedback kontinuerligt
Iterativ proces - byg, test, revidér
Workshops og JAD-sessioner (Joint Application Development)

3. Construction (Konstruktion)

Konverter prototyper til funktionelle systemer
Brug af RAD-værktøjer og 4GL (Fourth Generation Languages)
Automatiseret kodegenerering hvor muligt
Genbrugelige komponenter
Kontinuerlig integration og test

4. Cutover (Implementering)

Test af komplet system
Konvertering af data
Brugeruddannelse
Overgang til produktion
Mindre omfattende end traditionelle metoder pga. løbende test

---

Fordele ved RAD

Hurtigere time-to-market - Produktet færdigt på kortere tid  
Reduceret risiko - Problemer opdages tidligt gennem prototyper  
Højere brugertilfredshed - Brugere involveret gennem hele processen  
Fleksibilitet - Let at tilpasse til ændrede krav  
Bedre kommunikation - Tæt samarbejde mellem alle parter  
Lavere omkostninger - Færre fejl og omkostninger ved ændringer

---

Ulemper ved RAD

Kræver dedikerede ressourcer - Brugere skal kunne afsætte tid  
Ikke egnet til store systemer - Fungerer bedst ved små-mellemstore projekter  
Teknisk risiko - Kræver erfarne udviklere og gode værktøjer  
Skalerbarhed - Kan give problemer med performance/skalerbarhed  
Mindre dokumentation - Kan være problematisk ved vedligeholdelse  
Afhængig af modularisering - System skal kunne opdeles i komponenter

---

Hvornår bruge RAD?

**RAD er velegnet når:**

Projektet skal færdiggøres hurtigt
Budgettet er begrænset
Kravene kan ændre sig
Systemet kan opdeles i moduler
Brugere kan deltage aktivt
Erfarne udviklere er tilgængelige

**RAD er IKKE velegnet når:**

Systemet er meget komplekst eller stort
Høj performance er kritisk
Systemet har mange tekniske risici
Brugere ikke kan deltage aktivt
Omfattende dokumentation er påkrævet
###### Systemet skal integreres med mange legacy-systemer
---

**Værktøjer til RAD**

Low-code/no-code platforme (OutSystems, Mendix, PowerApps)
4GL værktøjer (Visual Basic, PowerBuilder)
Prototype-værktøjer (Figma, Adobe XD)
Komponentsbiblioteker og frameworks
Visuelle udviklings-IDE'er

---

**Konklusion**

RAD er ideel til projekter hvor hastighed og fleksibilitet er vigtigere end omfattende dokumentation og lange planlægningsfaser. Metoden kræver dog stærkt engagement fra brugere og erfarne udviklere for at lykkes.
